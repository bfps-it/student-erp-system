  Fills form: Name, Phone, Email, Position, Experience, Qualification, Resume (upload), Cover Letter

SECTION 3: CAREER DUPLICATE DETECTION AND MERGE LOGIC



3.1 Duplicate Detection Keys

Two career applications are considered DUPLICATES if ALL of these match:

  Key 1: applicantEmail (normalized to lowercase, trimmed)

  Key 2: positionTitle (normalized to lowercase, trimmed)

  Key 3: Both submitted within 72 hours of each other



If the applicant applies for TWO DIFFERENT positions, those are NOT duplicates.

They are two separate valid applications and both appear in the admin panel.



3.2 Detection Algorithm

TRIGGER: New career application arrives (from either channel)



STEP 1: Normalize incoming data

  - applicantEmail: toLowerCase().trim()

  - positionTitle: toLowerCase().trim()



STEP 2: Query database

  SELECT * FROM CareerApplication

  WHERE LOWER(email) = [normalized email]

  AND LOWER(positionTitle) = [normalized position]

  AND isDuplicate = false

  AND status != 'WITHDRAWN'

  AND createdAt >= NOW() - INTERVAL '72 hours'



RESULT A: No match found

  -- Create new CareerApplication record normally

  -- source = FORM | EMAIL (whichever channel)

  -- isDuplicate = false

  -- Auto-reply email to applicant: 'We have received your application for [Position].'

  -- Notification to admin panel: new application badge +1



RESULT B: Existing application found (DUPLICATE DETECTED)

  -- PRIMARY: the existing application (created first)

  -- SECONDARY: the incoming new submission



  Action on PRIMARY:

    -- source updated to MERGED (was FORM or EMAIL, now MERGED)

    -- If SECONDARY has resume and PRIMARY does not: attach SECONDARY resume to PRIMARY

    -- If SECONDARY has cover letter and PRIMARY does not: append to PRIMARY

    -- Add dedup log: 'Merged [EMAIL/FORM] submission received at [timestamp]'

    -- PRIMARY stays in admin queue unchanged



  Action on SECONDARY:

    -- Create record with isDuplicate = true

    -- mergedIntoId = PRIMARY application ID

    -- status = DUPLICATE_CANCELLED

    -- Do NOT show in admin panel main list

    -- Do NOT send new application notification to admin



  Auto-reply to applicant:

    'Thank you for your application for [Position] at BFPS.

     We had already received your application (Ref: [primaryId]).

     Your submission has been noted and combined with your existing application.

     We will contact you at this email address regarding next steps.'



3.3 Career Application Prisma Schema

model CareerApplication {

  id                 Int                    @id @default(autoincrement())

  referenceId        String                 @unique @default(cuid())

  jobPostingId       Int?

  jobPosting         JobPosting?            @relation(fields: [jobPostingId], references: [id])

  applicantName      String

  email              String

  phone              String?

  position           String                 -- normalized position title

  experience         String?

  qualification      String?

  resumeUrl          String?                -- Cloudinary URL

  resumeCloudinaryId String?

  coverLetter        String?                @db.Text

  source             ApplicationSource      -- FORM | EMAIL | MERGED

  status             ApplicationStatus      -- NEW | REVIEWING | SHORTLISTED | INTERVIEW | SELECTED | REJECTED | WITHDRAWN | DUPLICATE_CANCELLED

  isDuplicate        Boolean                @default(false)

  mergedIntoId       Int?

  deduplicationLog   String?                @db.Text

  autoReplySent      Boolean                @default(false)

  adminNotes         String?                @db.Text

  reviewedBy         Int?

  createdAt          DateTime               @default(now())

  updatedAt          DateTime               @updatedAt

}



enum ApplicationSource { FORM EMAIL MERGED }

enum ApplicationStatus { NEW REVIEWING SHORTLISTED INTERVIEW SELECTED REJECTED WITHDRAWN DUPLICATE_CANCELLED }



Indexes required:

  @@index([email, position])

  @@index([status])

  @@index([isDuplicate])

  @@index([createdAt])

PARSING LOGIC FOR EACH INCOMING EMAIL:

  1. Extract senderEmail: from.value[0].address

  2. Extract senderName: from.value[0].name

  3. Extract subject: subject field

  4. Extract body: text or html field (convert HTML to plain text)

  5. Extract position from subject using keywords:

     - Look for patterns: 'Application for [X]', 'Applying for [X]', 'Re: [X]'

     - If subject matches an active JobPosting title: link to that posting

     - If no match: position = subject string (admin can correct from panel)

  6. Check for resume attachment:

     - Accepted MIME types: application/pdf, application/msword,

       application/vnd.openxmlformats-officedocument.wordprocessingml.document

     - If valid attachment found: upload to Cloudinary, store URL

     - If attachment is invalid type: log warning, proceed without resume

  7. Run DUPLICATE DETECTION (Section 3.2)

  8. Create CareerApplication record (or merge per dedup logic)

  9. Mark email as READ

  10. Send auto-reply from noreply@bfpsedu.in



UNRECOGNIZED EMAIL HANDLING:

  If email appears to be spam (no name, no readable content): mark READ, skip

  If email has no position context: create application with position='Unspecified'

  Admin can update position and link to job posting from the panel

SECTION 7: CONTACT DUPLICATE DETECTION AND MERGE LOGIC



7.1 Duplicate Detection Keys for Contact

Two contact enquiries are considered DUPLICATES if:

  RULE A: Same email address AND submitted within 2 hours of each other

  OR

  RULE B: Same phone number AND submitted within 2 hours of each other

  OR

  RULE C: Same email AND same subject (normalized) regardless of time window



ANY ONE of Rule A, B, or C triggers duplicate detection.



7.2 Contact Detection Algorithm

TRIGGER: New contact enquiry arrives (form or email)



STEP 1: Normalize

  - email: toLowerCase().trim()

  - phone: remove spaces, dashes, +91 prefix if present, keep 10 digits

  - subject: toLowerCase().trim()



STEP 2: Query for duplicates

  SELECT * FROM ContactEnquiry

  WHERE isDuplicate = false

  AND status != 'RESOLVED'

  AND (

    (email = [email] AND createdAt >= NOW() - INTERVAL '2 hours')

    OR (phone = [phone] AND phone IS NOT NULL AND createdAt >= NOW() - INTERVAL '2 hours')

    OR (email = [email] AND LOWER(subject) = [normalized subject])

  )



RESULT A: No match -- Create new ContactEnquiry, source = FORM or EMAIL

  Auto-reply: 'Thank you for contacting BFPS. We will respond within 24 hours.'



RESULT B: Match found -- DUPLICATE DETECTED

  PRIMARY: existing record

  SECONDARY: incoming



  Action on PRIMARY:

    -- source updated to MERGED

    -- If SECONDARY message has additional info not in PRIMARY: append to PRIMARY notes

    -- Dedup log updated with timestamp and channel of secondary



  Action on SECONDARY:

    -- Record created with isDuplicate = true

    -- mergedIntoId = PRIMARY ID

    -- status = DUPLICATE_CANCELLED

    -- NOT shown in admin panel



  Auto-reply to sender:

    'Thank you for reaching out to BFPS.

     We have already received your enquiry (Ref: [primaryRefId]).

     Our team will contact you shortly. No further action needed.'

PARSING LOGIC FOR EACH INCOMING EMAIL:

  1. Extract: senderEmail, senderName, subject, body (plain text)

  2. Filter out: auto-replies, bounce messages, newsletter subscriptions

     - Skip if subject contains: 'auto-reply', 'out of office', 'delivery failed', 'mailer-daemon'

     - Skip if sender is: noreply@*, no-reply@*, mailer-daemon@*

  3. Extract phone from body using regex: Indian phone patterns

     Pattern: /(?:\+91|91|0)?[6-9]\d{9}/

  4. Run DUPLICATE DETECTION (Section 7.2)

  5. Create ContactEnquiry record or merge

  6. Mark email as READ

  7. Send auto-reply from noreply@bfpsedu.in

  8. Notify admin panel: new enquiry badge +1



8.1 Important: ADMISSION ENQUIRY vs CONTACT ENQUIRY

Admission enquiries go to: admission@bfpsedu.in (handled by Lead CRM module)

General contact goes to: info@bfpsedu.in (handled by Contact Enquiry module)

These are SEPARATE modules. Do NOT mix them.

  leads table         <- admission enquiries (CRM pipeline)

  contact_enquiries   <- general contact (info/support requests)

  career_applications <- job applications
